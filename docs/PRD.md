# 交易黑板 MVP PRD

状态：Draft v0.1  
产品代号：Codex Market Board / 交易黑板  
目标：让用户通过 Codex 完成“发现商品 → 比较 → 联系商家 → 下单 → 付款 → 产生可信成交数据”的最小闭环。

## 1. 产品定义

交易黑板不是传统电商首页，也不是匿名暗网市场。它是一个公开、结构化、可审计的交易目录：

- 商家公开注册资料和经营地区。
- 商品只公开标准化摘要、价格、库存状态、配送范围和合规状态。
- 完整咨询、个性化需求和议价通过 Codex 与商家代理沟通。
- 下单和支付生成不可由商家随意伪造的成交记录。
- 排名以真实支付、退款、争议、复购和履约为主要信号，不以好评数量为核心。

一句话体验：**告诉 Codex“我想买一把电动牙刷”，Codex 从黑板查询、解释筛选依据、联系商家并生成可核验的付款请求。**

## 2. 非目标

MVP 不做：

- 保存银行卡号或代替持牌支付机构清算资金；
- 承诺覆盖世界所有法律或自动保证商品合法；
- 文件、图片或私密聊天内容永久公开；
- 付费竞价排名、刷单奖励或简单五星好评榜；
- 全自动替用户付款；付款前必须展示商家、商品、金额、币种和退款条件并由用户确认。

## 3. 角色

### 买家

通过 Codex 搜索、比较、询价、确认订单和打开付款页；可以发表文本评论并查看真实成交统计。

### 商家

注册公开资料、声明经营地区和适用规则、发布商品摘要与图片、接受咨询、创建报价并履约。

### 平台审核员

维护公开规则、处理举报、屏蔽商品、冻结交易入口并发布带原因码的审核决定。

### Codex 代理

只调用结构化工具：搜索、读取详情、创建报价、创建订单、获取支付链接、查询订单。代理不能绕过付款确认或合规屏蔽。

## 4. 核心流程

### 4.1 商家入驻

1. 提交显示名、主体类型、经营国家/地区、联系方式验证状态。
2. 接受适用市场规则版本，并声明受限品类与许可证信息。
3. 平台生成商家 ID 和公开档案；敏感验证材料不进入公开黑板。
4. 未验证商家可以保存草稿，但不能收款。

### 4.2 发布商品

1. 商家填写标准字段与图片。
2. 系统按销售地、商品分类和规则版本执行合规检查。
3. 状态为 `draft / review / active / restricted / blocked / archived`。
4. 只有 `active` 商品可被搜索和创建订单；屏蔽记录公开原因码和规则版本，不公开敏感证据。

### 4.3 Codex 购买

1. 用户说：“我想买电动牙刷，预算 300 元以内。”
2. Codex 调用搜索接口，展示候选项及筛选依据。
3. 用户继续按价格、配送、成交量、退款率等过滤。
4. 用户选择商品后，Codex 可向商家发送问题或请求报价。
5. Codex 展示最终订单摘要；用户明确确认后创建订单。
6. 平台向支付服务商创建托管/收款会话，返回官方支付页或经验证的二维码。
7. 只有服务商 webhook 验证成功后，订单才记为已付款并计入成交统计。

### 4.4 评论与“用脚投票”

- 只有关联真实订单的账号可发布“已成交评论”；其他评论标记为“未验证讨论”。
- 不展示简单综合星级，不用评论数量直接排名。
- 公开成交数、退款率、争议率、按时履约率和数据时间窗。
- 取消、退款、拒付和被判定刷单的订单不贡献正向排名。

## 5. 公开黑板字段

### 商家 `merchant`

| 字段 | 公开 | 说明 |
|---|---:|---|
| `id`, `display_name`, `entity_type` | 是 | 稳定标识和主体类型 |
| `operating_regions` | 是 | 经营国家/地区 |
| `verification_level` | 是 | 未验证/基础/增强 |
| `policy_acceptances` | 是 | 接受的规则 ID 与版本 |
| `joined_at`, `status` | 是 | 入驻时间与状态 |
| 身份证件、银行账户、真实地址 | 否 | 仅合规服务保存 |

### 商品 `listing`

公开：`id`、商家 ID、标题、分类、摘要、结构化属性、价格区间、币种、库存状态、配送范围、图片、合规状态、规则版本、创建/更新时间。  
不公开：买卖双方聊天、内部审核证据、个人收货信息。

### 成交统计 `trade_stats`

公开聚合数据：已付款订单数、完成订单数、退款数、争议数、复购买家数、统计时间窗、最后更新时间。小样本数据应显示样本量，不制造虚假精确度。

### 规则 `policy`

公开：规则 ID、适用地区、版本、生效时间、来源链接、受限类别、平台执行动作和变更记录。

## 6. 排名

默认排序必须可解释，并允许用户显式选择价格、最新、成交量等排序方式。

基础可信分只使用可核验事件：

```text
score = confidence(sample_size)
      × (completed_orders + repeat_buyers × 2)
      × fulfillment_rate
      × (1 - refund_rate)
      × (1 - dispute_rate × 2)
```

约束：

- 每个指标展示时间窗和样本量；
- 同一付款工具、设备、地址或异常关联交易进入风控，不立即计分；
- 商家不能购买自然排名；赞助内容若未来存在，必须独立区域并明确标识；
- 新商家通过“新上架”入口获得曝光，不伪造可信分。

## 7. 合规与治理

- 商家声明不是免责条款，也不能使违法商品自动合法。
- 平台按销售地和买家所在地选择规则集合；无法判断时进入人工审核。
- 商品被举报或规则变更后，可立即停止新订单，但历史订单与审计事件不可静默删除。
- 每个屏蔽决定包含 `reason_code`、规则版本、时间和申诉状态。
- 严禁品类在产品上线前必须形成明确清单；不同地区的限制分开维护。
- 隐私、消费者保护、税务、退款和支付牌照需要上线国家的专业法律审核。

## 8. 支付

统一接口 `PaymentProvider`：

- `createCheckout(order)`
- `verifyWebhook(request)`
- `getPaymentStatus(reference)`
- `refund(reference, amount)`

首个实现建议使用托管结账页，平台不接触卡号。二维码支付也必须来自受支持的支付服务商并绑定订单、金额、币种和过期时间；不接受商家上传任意个人收款码作为可信支付凭证。

订单状态：

```text
draft → awaiting_buyer_confirmation → awaiting_payment
      → paid → accepted → fulfilled → completed
      ↘ cancelled / expired / refunded / disputed
```

## 9. Codex 工具集（最小完整集）

| 工具 | 用途 |
|---|---|
| `market.search` | 按自然语言和结构化条件搜索 |
| `market.get` | 读取商品、商家、规则和统计 |
| `market.ask` | 联系商家或请求报价 |
| `market.order.preview` | 生成不落单的订单摘要 |
| `market.order.create` | 用户确认后创建订单 |
| `market.pay` | 获取支付服务商页面/二维码 |
| `market.order.status` | 查询付款与履约状态 |

商家侧 MVP：`merchant.register`、`listing.create`、`listing.update`、`order.respond`。

## 10. MVP 验收标准

1. 一个商家可以注册并发布一个合规状态为 active 的商品。
2. Codex 能搜索电动牙刷并按价格、配送地和真实成交指标筛选。
3. 用户确认前不会创建付款会话。
4. 伪造客户端“已付款”不能改变订单；只有验证过的 webhook 可以。
5. 完成、退款和争议事件会重新计算公开统计与排名。
6. blocked 商品不能创建新订单，且黑板显示原因码和规则版本。
7. 评论不能直接提高排名，已成交评论具有可验证标记。
8. 私密聊天、地址和支付凭证不会出现在公开接口。

## 11. 分阶段实施

### Phase 1：可信目录

商家、商品、规则、公开搜索、合规状态、图片引用、可解释排名。

### Phase 2：真实成交

报价、订单状态机、单一支付服务商、webhook、退款/争议事件、成交统计。

### Phase 3：Codex 交易体验

Codex skill/MCP 工具、自然语言筛选、订单确认卡片、商家对话桥接。

### Phase 4：治理扩展

多地区规则、申诉、更多支付适配器、异常交易检测和公开透明度报告。

## 12. 待决定事项

- 首发国家/地区和首个允许销售的商品分类；
- 平台是纯目录、撮合方还是交易平台主体；
- 首个支付服务商及商家收款账户模型；
- 履约、退款和争议由谁裁决；
- Codex Ping 消息是否只做咨询，还是承载经过签名的报价引用。
